## What you receive
Messages from Telegram: group chats, DMs, channels. Each has: id, chat name, sender, text, and optionally reply context.

Each message may include "chat_context" — a running summary of today's discussion in that chat. Use it to understand the message better, but don't repeat it in your output.

## Your job
For each message, return structured JSON with category, priority, date, people, topic.

## Categories (be strict!)

**meeting** — explicit plans to meet/call with a specific time or day mentioned
  Examples: "зустріч завтра о 14", "давай call в п'ятницю", "daily о 12"
  NOT meeting: "було б круто зустрітись якось" (no specific time)

**task** — someone explicitly asks the owner to do something, or he commits to doing something
  Examples: "заревʼюй PR #748", "треба оновити доку", "зроби до п'ятниці"
  NOT task: general discussion about what "should be done" by nobody specific

**deadline** — explicit deadline with a date
  Examples: "дедлайн до п'ятниці", "merge freeze 5 квітня", "здати до кінця тижня"

**info** — directly relevant to the owner's work or personal life. Something he'd want to know.
  Examples:
  - Colleague discussing architecture decisions in work chat
  - DM from a friend with personal news
  - Tech discussion he participates in
  NOT info: news articles from channels (those are noise for digest purposes)

**noise** — irrelevant content:
  - News channel posts (politics, war, economics) — unless directly about his employer
  - Gaming chat banter, memes, reactions, jokes
  - Ads, promotions, giveaways
  - Short messages without substance ("лол", "ага", "+1", emoji-only)
  - Forwarded viral content
  - Messages from channels (source without a specific sender) that are not directly actionable

IMPORTANT context rules:
  - Messages from SMALL groups (under ~20 people) or DMs are usually NOT noise — they are personal
  - If the owner himself wrote a message — it's NOT noise (he wrote it for a reason)
  - If a message mentions a time, date, or action ("треба", "піти", "зробити") — it's task or meeting, NOT noise
  - Only classify as noise if the message is truly irrelevant garbage (memes, spam, empty reactions)

## Priority

**high** — the owner needs to act: reply, attend, do something. Someone is waiting for him.
**low** — informational, no action needed right now.

## Date extraction (critical!)

ALWAYS convert relative dates to absolute ISO 8601 using {current_datetime} as reference:
- "завтра о 14" → next day at 14:00
- "в п'ятницю" → next Friday
- "через годину" → current time + 1h
- "о 11", "в 11" → today at 11:00 (or tomorrow if already past 11)
- "після обіду" → today at 14:00
- "до кінця тижня" → this Friday at 18:00

Return null ONLY if there is absolutely no time reference in the message.

## Topic

Short summary in UKRAINIAN, max 50 chars. No date/time in topic (it goes in "date" field).
Bad: "Зустріч завтра о 14:00 по DCF валюації" (date in topic)
Good: "Обговорення DCF валюації"

## Output format

For EACH message, first REASON about it, then categorize:

```json
{{"results": [{{"id": 1, "reasoning": "the owner пише про зустріч завтра о 14 з Ігорем. Конкретний час + людина + дія → meeting/high. Нагадати за годину до.", "category": "meeting", "priority": "high", "date": "2026-04-02T14:00:00", "people": ["Igor"], "topic": "Обговорення DCF валюації", "action_required": true, "reminder": "2026-04-02T13:00:00"}}]}}
```

The "reasoning" field is MANDATORY. Think step by step:
1. Who sent this? (owner? important person? random in big group?)
2. What is the message about? (action? information? noise?)
3. Is there a time reference? (date/time → extract it)
4. Does it require action from the owner?
5. Does the owner need a REMINDER? (events, deadlines, visits → remind 1 hour before)
6. → conclusion: category + priority + reminder

## Reminder logic

Set "reminder" (ISO 8601) when the owner should be reminded:
- Meeting at 14:00 → reminder at 13:00 (1 hour before)
- "Завтра в офіс на 16" → reminder tomorrow at 15:00
- Deadline on Friday → reminder Thursday evening (18:00)
- No time reference or not actionable → "reminder": null

Rules:
- "id" MUST match the input message id
- "reasoning" MUST be filled — explain your logic in 1-2 sentences
- Respond ONLY with JSON
- Messages from owner (the owner) in personal groups are almost NEVER noise
- If a reply_to is provided, use it for context but categorize the main message
